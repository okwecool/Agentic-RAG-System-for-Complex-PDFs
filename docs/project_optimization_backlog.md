# 项目整体待优化文档

## 1. 文档目的

本文档用于沉淀当前项目在实现层面已经暴露出的系统性优化点，帮助后续开发时区分：
- 哪些是当前已落地的保守逻辑
- 哪些是已经确认存在的瓶颈
- 哪些优化值得进入下一阶段

本文档只记录优化方向，**当前逻辑先保留不动**。

---

## 2. 当前总体判断

项目目前已经具备：
- 可运行的 PDF ingestion 主链路
- 可落盘的 parsed / chunk 中间产物
- 可运行的 chunk 异常巡检
- 一批基于真实 PDF 样本迭代出的通用清洗规则

但从最近几轮异常文档排查可以确认：

1. 复杂 PDF RAG 的主要瓶颈还在版面分析层
- parser / cleaner / section_builder 的上游质量，直接决定 chunk 质量

2. chunk 问题很多不是 `chunk_size` 导致
- 更常见的是 layout、阅读顺序、表格识别、侧栏识别、页眉页脚处理的问题

3. 当前实现适合继续保守演进
- 先保留当前逻辑
- 在文档层面明确瓶颈和后续路线
- 避免为了个别样本过早重构

---

## 3. 当前版面分析的基础策略

当前版面分析本质上是一套“轻量启发式 layout parsing”，并不是完整的页面理解系统。

当前主链路可以概括为：

1. 使用 `PyMuPDF get_text("dict")` 提取原始 text blocks
- 直接依赖 PDF 引擎给出的 block、line、span、bbox

2. 基于启发式规则判断 block 类型
- `heading`
- `paragraph`
- `list_item`

3. 在 `cleaner` 中做块级清洗与局部重组
- 重复 header / footer 过滤
- 图表页短数字和图例过滤
- 跨行正文合并
- 可疑 heading 降级

4. 在 `section_builder` 中用 heading 维护 `section_path`

5. 在 `chunker` 中按 `section_path + page_no` 聚合切片

这套策略的优点：
- 简单
- 快
- 容易加规则
- 适合当前 Phase 1 快速迭代

这套策略的缺点：
- 强依赖原始 block 质量
- 缺少显式页面分区能力
- 缺少阅读顺序重建
- 缺少文档级版面自适应

---

## 4. 当前已经暴露出的系统性问题

### 4.1 当前是 block-first，不是 layout-first

当前实现默认：
- PDF 引擎给出的 block 基本可用
- 页面阅读顺序接近 block 顺序
- 封面、侧栏、图表页、双栏页可由后处理修正

这在简单 PDF 上有效，但在复杂版面上会失效。

典型表现：
- 左侧栏被当成正文
- 图表标题、图例、来源混入正文 chunk
- 同一页多栏内容交错进入同一 section

### 4.2 标题识别仍偏启发式

虽然这几轮已经收紧了 heading 规则，但当前判断仍主要依赖：
- 行数
- 字号
- bold 比例
- 编号样式

缺点是：
- 对模板差异敏感
- 还没有利用“文档内部字体分布”做相对判断

### 4.3 表格识别还缺少“整表视角”

当前表格识别基于 `page.find_tables()`，再叠加过滤。

已经解决了：
- 单列说明框误识别
- 联系人框误识别
- 页脚页码框误识别
- 装饰性窄长表格误识别

但仍存在：
- 一张表被拆成多张小表
- 表内残留文字块未完全去除
- 表格与说明文字、图题之间缺少绑定

### 4.4 页面的区域概念还不够明确

当前没有显式区分：
- 主正文区
- 侧栏区
- 页眉区
- 页脚区
- 图表区
- 封面元信息区

于是大量规则只能在 block 层“猜”，而不是先分区再处理。

---

## 5. 已确认有效、但仍应保留为当前逻辑的优化

最近几轮已经沉淀出一些有效的通用修复，这些应继续保留：

### 5.1 跨行正文合并

适用于：
- 正文被 PDF 按行拆散
- 连续同列文本被错误分成多个 block

收益：
- 避免单段正文碎片化
- 降低误判为 heading 的概率

### 5.2 更保守的 heading 判定

适用于：
- 大字号单行正文
- 封面标签
- 侧栏短标题

收益：
- 减少 `section_path` 被长句正文不断重置

### 5.3 页眉 / 页脚位置感知过滤

适用于：
- 重复页眉页脚
- 页脚声明
- 页码模式

收益：
- 减少 footer 污染正文和 chunk

### 5.4 图表页噪声过滤

适用于：
- 轴标签
- 图例
- 短数字
- 装饰条

收益：
- 对封面页和图表页效果明显

### 5.5 伪表格过滤

适用于：
- 单列长文本说明框
- 联系人 / 信息框
- footer page number table

收益：
- 避免 table 通道被大量非结构化内容污染

## 6. 版面分析的主要优化空间

这部分是当前最值得关注的系统级优化方向。

### 6.1 页面分区

当前最缺的是显式的 page zoning。

建议后续引入页面区域概念：
- `main_body`
- `sidebar`
- `header`
- `footer`
- `table_region`
- `chart_region`
- `cover_meta`

收益：
- 能先判断块属于哪个区域，再决定如何处理
- 比单纯在 `cleaner` 里叠加规则更稳

### 6.2 阅读顺序重建

当前阅读顺序过度依赖 block 原始顺序。

建议后续考虑：
- 双栏重排
- 左侧栏与右正文拆分
- 依赖 `x/y` 对齐关系重构阅读顺序

收益：
- 避免侧栏内容插进正文 chunk
- 避免多栏页面 section 混乱

### 6.3 文档内相对字体建模

当前标题规则还是绝对阈值为主。

建议后续加入文档级统计：
- 正文字号主峰
- 常见粗体层级
- 标题字号层级

收益：
- 标题判断更稳
- 模板适配性更好

### 6.4 表格整表合并与区域去重

建议后续做两件事：

1. 同页碎表合并
- 按列对齐、x 范围、y 连续性合并相邻 table

2. table 区域文本去重
- 如果表格块已存在，尽量去掉区域内重叠 paragraph

收益：
- 避免一张表切成多张小表
- 避免 table 和表内文本双重入库

### 6.5 页面模板识别

建议后续尝试做 page type classification：
- 封面页
- 正文页
- 大表格页
- 风险提示页
- 法律声明页

收益：
- 不同类型页面可以启用不同清洗和 chunk 策略

## 7. 结合真实文档得到的典型结论

### `doc_0f5cda2889f1`

说明：
- 当前策略对“单行拆散正文”比较敏感
- 说明框也容易误入 table 通道

结论：
- 跨行正文合并和伪表格过滤是必要保底能力

### `doc_3eb3bc009cd2`

说明：
- 误判 heading 会导致整份文档 chunk 爆炸

结论：
- heading 识别必须保守
- `section_path` 是 chunk 质量的核心依赖

### `doc_4a9bb6d9bafa`

说明：
- 封面 / 图表页会产生大量噪声块

结论：
- 页面区域和噪声过滤必须前置到 chunk 之前

### `doc_e15effe07d4e`

说明：
- 联系人框、页码框、装饰条都会污染 table 通道

结论：
- table extractor 需要允许“回退为非表格”

### `doc_b6cad8ed120d`

说明：
- 首屏是典型侧栏 + 正文混排
- 第 5 页是一页多表 / 表格碎裂问题

结论：
- 当前最缺的是页面分区和阅读顺序重建
- 这类问题不是 `chunk_size` 调参能解决的

---

## 8. 当前不建议立刻做的事

为了控制节奏，当前阶段不建议马上做这些重改：

1. 全面重写 parser
- 当前仍应先保留现有主链路，继续积累样本规律

2. 过早引入重量级 layout model
- 现在异常类型还在快速收敛，先把启发式边界摸清更划算

3. 用大量单文档模板规则覆盖问题
- 容易让 `cleaner` 和 `table_extractor` 变得不可维护

4. 盲目调整 `chunk_size`
- 这只能改变症状，不会解决版面根因

---

## 9. 推荐优化顺序

如果后续开始优化版面分析，建议顺序如下：

1. 页面分区
- 先建立 `main_body / sidebar / footer / table_region` 的概念

2. 阅读顺序重建
- 尤其针对双栏、侧栏和封面页

3. 文档内相对字体建模
- 替代部分绝对字号阈值

4. 表格整表合并
- 减少碎表和重叠文本

5. 页面模板识别
- 提高不同页面类型下的稳定性

---

## 10. 当前结论

当前逻辑应继续保留，因为它已经具备：
- 可运行
- 可调试
- 可巡检
- 可通过真实样本持续改进

但从系统视角看，项目的下一阶段优化重点已经比较明确：
- 不是继续堆更多局部规则
- 而是逐步从 block-first 过渡到更稳定的 layout-aware 策略

一句话总结：

当前系统已经能工作，但还没有真正“理解页面”；后续最值得投入的优化，是页面分区、阅读顺序重建和表格整表视角。
