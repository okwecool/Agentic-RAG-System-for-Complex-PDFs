## 1. 当前实现到了什么程度

从当前代码和实际运行链路来看，这个项目已经不再是“空骨架”状态，而是一个**基础主链可运行的 RAG 系统**。

当前已经具备的能力包括：

- PDF 解析、清洗、切片
- chunk 落盘与批量处理
- BM25 + 向量召回 + hybrid fusion
- 可插拔的 reranker 架构
- 可插拔 reranker 架构已接入，并完成本地模型 smoke test
- CLI 问答
- HTTP API 问答
- OpenAI 兼容 LLM 调用链
- 基础 evidence 与 citation 返回
- 一定程度的结果去重、局部聚合与通用排序增强

因此，从工程阶段判断，当前系统已经可以定义为：

**“基础 RAG 主链已完成，但质量增强层仍未闭环。”**

---

## 2. 还没有真正实现的关键能力

虽然主链已经通了，但如果按“一个真正可用的最小化 RAG 系统”来衡量，当前仍然缺少几项关键能力。

### 2.1 引用校验还比较弱

当前 `citation_auditor` 仍然是轻量级实现，本质上只是确认返回的 chunk 是否存在。

还没有做到：

- claim 级别的引用校验
- answer 句子与证据片段的精确对齐
- unsupported claim 识别
- 引用冲突检查

这意味着系统已经能“返回引用”，但还没有做到“证明答案中的每条结论都由引用支撑”。

### 2.2 上下文组装能力还不够强

当前 `ContextPacker` 仍然偏简单，主要是截取前几个检索结果。

还没有做到：

- token budget 控制
- 多文档上下文平衡
- 去冗余压缩
- query-aware 上下文挑选
- parent-child 回溯组装

这会导致一个问题：即使召回到了相关 chunk，最终给模型的上下文也不一定是最优组合。

### 2.3 chunk 质量仍然不够稳定

虽然前面已经修复了很多异常文档，但从整体看，chunk 层仍然存在以下问题：

- 图表碎片混入正文 chunk
- 图表轴标签、图例、短数字块仍会进入证据候选
- 有些 chunk 太碎
- 有些 chunk 又过大
- 图表页和表格页的 chunk 仍然不够适合作答

这说明当前 ingestion 和 chunking 虽然已具备可用性，但还没有达到稳定高质量阶段。

### 2.4 结构化证据还没有建立

目前系统本质上仍然是**以文本 chunk 为核心的 RAG**。

虽然表格和图表已经可以被一定程度地抽取和召回，但它们还没有真正变成可利用的结构化证据。

还缺：

- 表格行列结构
- 图表元数据
- 指标-时间-数值记录
- 面向精确问答的 structured evidence

如果后续需要稳定回答财务、销量、份额、增长率之类的问题，仅靠原始文本 chunk 很难足够稳定。

### 2.5 reranker 虽然接入了，但还没有成为默认质量闭环的一部分

目前 reranker 已经具备：

- 可插拔架构
- 可插拔本地模型接入能力
- 可配置 provider

但还没有真正完成：

- 默认主流程启用并评估
- 不同 query 下的效果对比
- 与 fusion 参数协同调优
- rerank 前后结果差异分析

所以目前它更多还是“能力已接入”，还不是“默认质量保障环节”。

### 2.6 缺少完整评测体系

当前系统已有很多实际运行验证，但仍然缺少系统化评测集。

还没有形成稳定评测体系去量化：

- chunk 质量
- 检索命中率
- rerank 提升
- 回答正确率
- 引用准确率

这意味着很多优化目前仍然依赖人工 spot check，而不是稳定指标驱动。

## 3. 当前最值得优先优化的方向

结合当前工程状态和问题暴露情况，最值得优先投入的方向如下。

### 3.1 让 reranker 真正进入默认检索闭环

这是当前性价比最高的一步。

原因：

- embedding 已经有
- hybrid retrieval 已经有
- 本地 reranker 模型已经接上

因此下一步最自然的是：

- 将 reranker 纳入默认实验链
- 对比 `noop` 与实际启用的本地 reranker
- 评估不同 fusion 参数与 rerank 的协同效果

这一步的收益是最直接的，因为它不需要重写 ingestion，却可能显著改善 top-k 质量。

### 3.2 做 parent-child chunking

这是当前最重要的结构升级之一。

它主要解决：

- 小 chunk 检索更准
- 大 chunk 回答更全

当前系统如果只靠单层 chunk，很难同时兼顾这两点。

parent-child 设计能让系统：

- 用 child 召回
- 用 parent 回答
- 保留细粒度引用
- 给生成模型更完整上下文

### 3.3 做表格 / 图表半结构化抽取

当前很多高价值信息明明在 PDF 里，但系统只能看到它的文本碎片。

后续至少需要逐步建立：

- 表格标题
- 列头 / 行头
- 单位
- 时间轴
- 图表标题
- 图表维度信息

哪怕第一阶段还不能完整恢复数值，也应该先把“弱结构化文本”升级成“半结构化对象”。

### 3.4 强化 citation 对齐

如果系统后续要对外可靠使用，仅仅“返回几个 chunk”是不够的。

后续至少要逐步做到：

- claim 级 citation
- answer 到 evidence 的显式映射
- 明确 unsupported claim

这会直接决定系统的可信度。

### 3.5 建立小规模评测集

当前如果没有评测集，后续很多优化都只能凭感觉判断。

建议先建立一小批问题：

- 覆盖正文型问题
- 覆盖表格型问题
- 覆盖图表型问题
- 覆盖跨页问题

这样后面每次改 chunk、fusion、reranker，都能快速做对比。

## 4. 当前实现的主要瓶颈在哪里

当前最主要的瓶颈不是：

- API 是否能调通
- LLM 是否能返回答案
- embedding 是否已经接入

这些主链问题基本都已经解决了。

当前的核心瓶颈在于：

### 4.1 证据质量

检索层拿到的 chunk 本身往往还不够“适合作答”。

### 4.2 证据表示方式

图表、表格、复杂页面中的高价值信息，目前仍主要以原始文本碎片形式存在。

### 4.3 证据选择能力

召回到了不代表能选对。当前系统的候选整理能力还不够强。

### 4.4 证据验证能力

模型已经能给答案，但系统还没有足够强的机制保证答案中的 claim 都可追溯、可验证。

---

## 5. 当前阶段的整体判断

当前阶段最准确的描述不是：

- “系统还没做起来”

也不是：

- “系统已经完成”

而是：

**“主链已成型，质量层仍需补齐。”**

更具体地说：

- ingestion 已经具备基础可用性
- retrieval 已经具备基础闭环
- generation 已经具备实际调用能力
- API 已经具备真实对外能力

但是：

- 证据质量还不够稳
- chunk 表示还不够强
- reranker 还未成为默认质量保障层
- citation 还未达到可严格信任的程度

---

## 6. 推荐的近期推进顺序

如果从当前状态继续推进到一个更可靠的最小化 RAG MVP，推荐顺序如下：

1. 对比实验并启用 reranker
- `noop` vs `local_transformers`
- 不同 fusion 参数组合

2. 做 parent-child chunking 设计与第一版实现

3. 做表格 / 图表半结构化抽取

4. 建立小规模评测集

5. 强化 citation 对齐

这个顺序的优点是：

- 不会推翻现有主链
- 每一步都能直接提升结果质量
- 能逐步把系统从“可运行”推进到“可稳定回答”

---

## 7. 结论

当前系统已经完成了最小化 RAG 的“主流程实现”，但还没有完成最小化 RAG 的“质量闭环”。

换句话说：

- **能跑**
- **能答**
- **能返回引用**

但还没有做到：

- **稳定召回高质量证据**
- **稳定利用图表与表格信息**
- **稳定保证答案与引用严格对齐**

后续优化重点不应该再放在“多加一个接口”，而应该集中在：

- 更好的 chunk 组织
- 更强的 rerank
- 更好的结构化证据
- 更可靠的 citation 校验
