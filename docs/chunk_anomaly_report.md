# Chunk 异常巡检报告

## 1. 巡检范围

本轮巡检基于以下产物：
- `artifacts/manifests/ingestion_summary.json`
- `artifacts/manifests/chunk_audit_report.json`
- `artifacts/parsed/*.json`
- `artifacts/chunks/*.json`

已覆盖当前仓库内已落盘的 PDF ingestion 结果，并结合异常文档做了定向复盘。

核心观察指标：
- `chunk_count / page_count`
- 页面级 `heading_ratio`
- 页面级短 chunk 密度
- `table` chunk 是否存在“单列长文本 / 信息框 / 页码框”误识别

---

## 2. 异常类型归类

当前 chunk 异常主要分为四类：

### A. 正文误判为 heading

特征：
- 同一页出现大量 `heading`
- `section_path` 被长句正文不断重置
- chunk 结果被切成大量单行或短句

根因：
- PDF 原始 block 被按单行拆散
- 标题识别对“大字号单行正文”过于敏感
- `section_builder` 将误判的 heading 当作新章节

已落地修复：
- 收紧 `PyMuPDF` parser 的 heading 判定
- 在 `cleaner` 中增加跨行正文合并
- 对长句、非结构化标题做 heading 降级

典型文档：
- `doc_0f5cda2889f1`
- `doc_3eb3bc009cd2`

### B. 封面 / 图表页噪声过多

特征：
- 第 1 页 chunk 数异常高
- 含大量轴标签、图例、装饰条、刻度数字
- 页面视觉上不是正文页，但被切成大量 `paragraph`

根因：
- 图表页短文本和装饰元素被当作正文 block
- 封面信息层级复杂，标题、副标题、图例混入同一页面

已落地修复：
- 过滤图表轴标签和短数字块
- 过滤左侧图例型小文本块
- 过滤窄长装饰条和封面竖排文本
- 强化页眉 / 页脚重复块过滤

典型文档：
- `doc_4a9bb6d9bafa`
- `doc_f446786dc9cf`

### C. 伪表格 / 信息框误识别

特征：
- 联系人框、说明框、相关研究列表被识别成 `table`
- `table` chunk 实际不包含结构化行列信息
- 表格内容更像说明文本或列表

根因：
- `page.find_tables()` 会把边框明显但非表格的信息框识别为 table
- 单列表格、联系人框、页码框缺少过滤

已落地修复：
- 过滤单列长文本说明框
- 过滤稀疏联系人 / 信息框
- 过滤页脚右下角页码小表格
- 过滤窄高装饰性 table

典型文档：
- `doc_0f5cda2889f1`
- `doc_e15effe07d4e`

### D. 页眉 / 页脚重复内容污染 chunk

特征：
- 多页重复出现相同短文本
- 页脚免责声明、页码、披露声明进入正文 chunk
- 同一模板 PDF 多页残留相同 footer

根因：
- 仅按文本内容无法稳定识别所有模板页脚
- 部分报告的页脚文案不完全重复，但位置和语义相近

已落地修复：
- 引入位置感知的 header / footer 重复过滤
- 将页脚声明过滤改为“位置 + 通用声明关键词 / 页码模式”
- 去掉特定文案硬编码，保留通用模式识别

## 4. 已沉淀的通用修复策略

这些修改已经从“单文档定向补丁”收敛成通用策略：

1. 结构优先于窗口
- 优先修正 block 质量和 `section_path`
- 不靠盲目调小 `chunk_size` 解决碎片问题

2. 先清洗版面噪声，再做切片
- 图表页短数字、图例、装饰条、页脚声明都应在切 chunk 前过滤

3. heading 识别必须保守
- 对长句、正文样式、非结构化单行文本优先降级为 `paragraph`

4. 表格识别必须允许“回退为非表格”
- 说明框、联系人框、页码框、单列长文本框不应强行进入 `table` 通道

5. 页脚规则只保留模式级逻辑
- 避免将具体模板文案写死到 `cleaner`
- 使用“页脚位置 + 通用声明关键词 + 页码模式”判断

## 6. 下一步建议

建议继续按以下顺序推进：

1. 优先处理最新 top 异常文档
- 重点看首屏碎块和正文被拆散的问题

2. 将异常文档复盘沉淀成更多通用规则
- 尽量避免面向单个 PDF 模板写特殊逻辑

3. 在进入索引层前，先稳定 chunk 质量
- 尤其是表格 chunk 和页脚噪声

4. 保持“修复后重跑单文档 + 重写 audit report”的闭环
- 确保每次规则更新都能被报告反映出来
